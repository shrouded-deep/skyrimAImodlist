## 3. Risks / watch items not already prominent in decisions.md

### 3.1 Synthesis `PipelineSettings.json` reverted with no identified cause
The W4ENB patcher's source path (task-0024's fix) reverted from the
correct GitHub path back to the broken local `F:\` path at least once
during the session, and was re-applied. **The actual mechanism causing
this was never identified** — unlike the MO2 profile/executable-config
reversions (which were traced to MO2's in-memory-state-on-exit behavior),
nothing touches `PipelineSettings.json` in the known script/task history.
This could recur silently. Worth a dedicated investigation if it happens
again, rather than just re-applying the fix a third time.

### 3.2 MO2 state-drift is a pattern, guardrails are not yet universal
Confirmed instances this session: Pandora's `Settings.json` (global,
not instance-scoped), MO2's own Executables config (ini edits don't
stick unless made through MO2's UI), and MLO2's `modlist.txt` line +
DLL (silently dropped, cause never fully confirmed — see task-0026's
result for the leading theories). `Mo2ProfileGuardrails.ps1` now
provides a shared `Assert-Mo2Closed` check, and task-0028 hardened two
scripts to *enforce* it (not just import it — enforcement was found
missing in one script even though it dot-sourced the module). **Any new
script written in the future that touches MO2 profile/config files must
be checked for this enforcement explicitly** — don't assume "uses the
guardrails module" means "actually obeys it."

### 3.3 DynDOLOD/LOD output is increasingly stale relative to current mod state
`DynDOLOD.esp`/`Occlusion.esp` were disabled (task-0025) then re-enabled
during the Lux rollback (task-0030) once their masters resolved again —
but the underlying LOD data has not been regenerated since well before
the Pandora migration, the MLO2 experiment, the WSU/True Light
add-then-remove cycle, or the Lux Via restoration. The longer the Tier 2
window is deferred, the larger the eventual regen/verification burden
grows, and there's a nonzero chance of subtle visual mismatches already
present that haven't been noticed because they're not being actively
tested against.

### 3.4 WSU/True Light/Placed Light removal — confirm fully clean, not just disabled
task-0030's rollback disabled these mods as part of restoring Lux. Worth
explicitly confirming they were either fully uninstalled or are cleanly
inert while disabled — i.e. no orphaned enabled files, no leftover
`LightPlacer` JSON conflicts with the restored Lux CS Light JSONs. This
wasn't a stated acceptance criterion of task-0030 and may be worth a
quick verification pass.

### 3.5 Frozen Lux-dependent outputs — confirm actually functional post-rollback
`Actually Brighter Lux Templates.esp`, `ANV_Lux Orbis - SB Windhelm
Entrance.esp`, `ANV_Lux - W4ENB - No Grass in Caves.esp`, and `ANV_Lux -
Volkihar Soundscape Overhaul - W4ENB.esp` were re-enabled during the Lux
rollback on the assumption that restoring Lux resolves their missing
masters. This was verified at the plugin/master level (0 missing masters
reported) but not re-confirmed via live in-game testing at Windhelm or
Volkihar specifically since the rollback. Worth folding into whatever
the next in-game test pass is, rather than assuming clean based on the
master-resolution check alone.

### 3.6 "Fresh save required after behavior-engine changes" — now documented, but easy to forget generally
This cost a full diagnostic cycle (task-0029/0031) before being traced to
a stale save, not a real Pandora defect. It's now documented in
`pandora-manual-run.md`. Worth remembering as a general principle beyond
just Pandora specifically: **any future work that touches behavior
generation, skeleton patching, or similar systemic runtime state should
default to testing on a fresh save first**, not as an afterthought once
something looks broken.

### 3.7 Approval-phrase discipline is a process fix, not a guarantee
The "Task-XXXX approved" rule (in `AGENTS.md`) was added after a real
incident (a vague "run the next task" got misread as approval for the
MLO2 execution). The rule closes the specific gap that caused that
incident, but it depends on the human remembering to use the phrase and
the agent remembering to require it — it's a process control, not a
technical guarantee. Worth periodically sanity-checking that this is
still being followed correctly, especially under a new agent session
that hasn't internalized the incident history the way this conversation
has.

## 4. Process notes worth preserving

- **Self-service task creation** is standing practice: agents should
  create their own follow-up task files rather than just describing them
  in prose, except for genuine strategic/scope decisions, which need
  human sign-off first.
- **Pre-flight cross-check**: before starting any task, check
  `tasks/completed/` (not just your own queue) for anything that might
  contradict the task's premise — this was added after the McmRecorder
  timing collision (task-0011 executed before task-0009's contradicting
  finding had been seen).
- **Status vocabulary**: `done`, `deferred`, `superseded` are now
  distinguished in `tasks/README.md` — use them correctly so "real work,
  wrong status" and "no work needed" don't get confused again (this
  happened with task-0028 and task-0031 respectively).
- **Reducing the human-relay bottleneck** is an explicit, ongoing goal.
  Prefer having Claude Code/Cursor self-direct off `AGENTS.md`'s Curation
  Direction and existing decisions.md history where reasonable, rather
  than requiring a human (or this chat) to draft every task.
