## Description

After task-0022's MLO2 install and subsequent **MO2 profile clobber** (modlist
line dropped, `MLO.dll` missing), session work on `install-wsu-stack.ps1` exposed
a systemic gap: scripts that text-edit `profiles/…/modlist.txt` (and related
profile files) had no shared **MO2-must-be-closed** preflight, no post-install
verification, and no profile snapshots for diffable history.

User sign-off ("let's add the guardrails") extended the requirement beyond WSU
install alone: **any** repo script that mutates MO2 profile state must use the
same pattern going forward — not a one-off hardening inside
`install-wsu-stack.ps1` only.

Triggered by task-0022 MLO2 regression analysis and task-0026 WSU install
readiness review.

## Acceptance criteria

- [x] Add shared `scripts/Mo2ProfileGuardrails.ps1` dot-sourced by profile-touching
      automation (Anvil instance paths, `Assert-Mo2Closed`, MLO2 health/repair,
      modlist block helpers, optional profile snapshot)
- [x] Add `scripts/verify-mlo2.ps1` — standalone preflight (`-Repair` restores
      `MLO.dll` from Nexus 160748 archive in Downloads)
- [x] Refactor `scripts/install-wsu-stack.ps1` to use guardrails: hard-stop if MO2
      running, verify/repair MLO2, smart FOMOD-aware extract + install health
      checks, post-install modlist + `MLO.dll` verification, write snapshot under
      `modlist/exports/wsu-stack-post-install-*`
- [x] Document guardrail workflow in `modlist/mlo2-manual-run.md` (§0.5 install +
      `verify-mlo2.ps1` usage)
- [x] Establish documented rule: **mutation** scripts writing `modlist.txt`,
      `plugins.txt`, or `loadorder.txt` must call `Assert-Mo2Closed` with
      unconditional throw (diagnostics may catch/report instead)
- [x] Bring later profile scripts into compliance when added (`rollback-lux-from-mlo2.ps1`,
      `fix-loadorder-post-0030.ps1` — see Result post-close note)

## Result

### What was done

1. **`scripts/Mo2ProfileGuardrails.ps1`** (new shared module):
   - `Assert-Mo2Closed` — detects `ModOrganizer` process; throws before profile edits
   - `Test-Mlo2Healthy` / `Ensure-Mlo2Healthy` / `Repair-Mlo2` — task-0022-style
     FOMOD flatten re-extract for `MLO.dll` + `MLO.ini` (`enableColorConsistency=false`)
   - `Ensure-ModlistBlockAfterAnchor` — insert lighting-stack lines after
     `+Base Object Swapper`; strip duplicates on re-run
   - `Write-ProfileSnapshot` — copy `modlist.txt` to `modlist/exports/<label>-<timestamp>/`
   - `Install-ModFromArchive` + layout helpers (`Get-ModContentRoot`, `Copy-Mo2ModTree`,
     `Test-ModInstallHealth`) — FOMOD-aware extract with marker ESP / path / file-count checks
   - WSU-stack installers added to same module as session continued:
     `Install-TrueLightForWsu`, `Install-DialForAnvil`, `Install-WsuForAnvil`,
     `Install-WsuPatchHubForAnvil` (mesh-only / FOMOD-specific verification)
   - Header comment (added during task-0030 reconciliation): mutation scripts must
     hard-fail on open MO2; diagnostics may catch/report

2. **`scripts/verify-mlo2.ps1`** (new):
   - Reports MO2 running / `MLO.dll` present / `+Modern Lighting Overhaul 2` in modlist
   - `-Repair` invokes `Ensure-Mlo2Healthy -Repair`

3. **`scripts/install-wsu-stack.ps1`** (refactored):
   - Dot-sources guardrails; `Assert-Mo2Closed` preflight; `Ensure-Mlo2Healthy -Repair`
   - Per-mod install health (incl. Dust mesh path check, True Light asset verification)
   - `Assert-WsuPostInstall` + `Write-ProfileSnapshot` postflight

4. **`modlist/mlo2-manual-run.md`** — §0.5 updated with close-MO2-first workflow and
   `verify-mlo2.ps1` commands.
