## Description

task-0006 (Armor/Weapon/ConstructibleObject/LeveledItem conflict audit)
found that `Lux Orbis.esp` wins 4 `LeveledItem` conflicts that are
unrelated to lighting: `GuardGear`, `CWSoldierImperialGear`,
`CWSoldierSonsGear`, `CWSoldierSonsGear1H`. Inspected the diff for
`GuardGear` (`100561:Skyrim.esm`): vanilla has 5 entries, Lux Orbis's
version has 4 — it removes the entry for 12x `SteelArrow`
(`01397F:Skyrim.esm`). Same pattern (an entry removed) shows in the field
diff for the other three.

This is **low severity** — a guard/soldier no longer getting a small
stack of steel arrows added to their leveled inventory is a minor balance
nuance, not a functional break (unlike the McmRecorder.esp navmesh/
persistent-reference finding in task-0009, which is higher-severity).

Context already gathered: `Lux Orbis.esp` (Nexus id 56095) is a large,
actively-maintained exterior lighting overhaul (5M+ downloads, 17,981
endorsements) whose FOMOD explicitly "includes many patches and
improvements for a lot of modded content." That makes a bundled
gear-list tweak plausible as a FOMOD-selected patch option — but it's
still an odd thing for a lighting mod to touch, and worth a quick
confirmation rather than assuming.

## Result

**Confirmed intentional — core Lux Orbis feature, not a separate FOMOD patch.**

The [Lux Orbis Nexus description](https://www.nexusmods.com/skyrimspecialedition/mods/56095)
(mod 56095) explicitly documents two relevant design choices in the main mod
description (not a hidden FOMOD option):

1. *"Guards levelled list has been edited so that they stop using torches
   all around, pretty much destroying all meshes nearby."*
2. *"Most civil war content has been reworked."*

The four `LeveledItem` edits (`GuardGear`, `CWSoldierImperialGear`,
`CWSoldierSonsGear`, `CWSoldierSonsGear1H`) — each removing an ammo entry
(e.g. 12× Steel Arrow) — align with preventing guards/soldiers from
equipping torches or arrow stacks that interfere with Lux Orbis exterior
lighting meshes. This is expected behaviour for a Lux Orbis install.

**No action needed.** Reverting would restore guard torch behaviour that
Lux Orbis deliberately removes. No patch task created (low severity; revert
undesirable).
