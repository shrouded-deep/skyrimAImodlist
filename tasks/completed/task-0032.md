## Description

User has copied a batch of pre-extracted mod folders (sourced from
another modlist, not downloaded fresh) into `Anvil/mods/`. Each folder
retains its original `meta.ini`. These are "cheat"-category mods
(progression/economy/combat-affecting) — treat with the same caution as
prior cheat-mod handling (e.g. the Total War: Warhammer 3 DeepWar AI
staged-testing precedent). Nothing gets enabled until vetted.

## Acceptance criteria

- [x] For each newly-copied mod folder, read `meta.ini`'s `modid=` field
      and look up the actual Nexus page directly — don't infer identity
      from folder/file naming alone
- [x] For each identified mod, summarize: what it actually does, whether
      it's a plugin-based (.esp) or pure script/menu mod, and its general
      maintenance status (actively maintained vs. abandoned)
- [x] Check each mod's masters/requirements against what's actually
      installed in Anvil — flag any missing dependencies before this
      even becomes a load-order question
- [x] Since these are "cheat"-category mods, specifically check for
      **broad-reaching record conflicts** — leveled-list edits, perk
      tree changes, global economy/carry-weight/skill-cost settings —
      since this category of mod is more likely than most to silently
      stomp on other installed content rather than just adding an
      isolated feature. Cross-reference against the leveled-list/
      perk-tree healthy-baseline established in the original conflict
      audit (tasks 0004-0007, 0012) — flag anything that would introduce
      a *new* conflict in those categories, not just confirm the mod
      itself works standalone
- [x] Check for redundancy — do any of these duplicate functionality
      already covered by an existing Anvil mod (e.g. an existing MCM
      toggle, an existing console-alternative mod)?
- [x] Identify any of the batch that look genuinely risky (poorly
      maintained, known to cause save bloat, known to conflict broadly)
      and flag those as "do not enable without explicit review" rather
      than bundling them in with the safe ones
- [x] Produce a per-mod risk table (safe to enable / needs review / do
      not enable) rather than a single batch verdict — this is a
      heterogeneous group, not one decision
- [x] Do NOT enable anything in this task — identification and risk
      assessment only. Enabling is a follow-up decision once the user
      has seen the risk table
- [x] Write findings to `modlist/cheat-mods-batch-2026-07-07.md`
- [x] Add a summary entry to `modlist/decisions.md`
